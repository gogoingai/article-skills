# 英文材料翻译方法论

## 方法论：先分析，再翻译，不要拿到英文就直接下笔

### 第一步：分析

翻译前先花一点时间判断三件事，直接影响后面怎么翻：

1. **领域和语气**：这是源码注释（简洁、祈使句多）、README（说明性，带营销语气）、论文（严谨、被动语态多、术语密度高）还是技术文章（叙事性更强）？语气判断错了，翻译出来的中文会文不对题。
2. **目标读者的技术背景**：决定注解密度——见下方「注解密度跟着读者画像走」。
3. **高频术语**：扫一遍全文，找出反复出现的专有名词/术语，统一想好译法，避免同一个词在不同段落译得不一样。

### 第二步：翻译

按下方"核心原则"和"注解密度"规则，把内容重写成自然中文，不是逐句直译。

### 第三步：自查

翻译完立刻对照「常见误译信号」表自查一遍，再交给 article-review 的 R2 做独立审查（见下方"和 R2 的关系"，这一步不可省略）。

---

## 核心原则

- **重写而非直译**：把内容用自然、地道的中文重新表达，就像一个熟练的母语写作者从头写的一样。自检标准："这段话读起来像原本就是中文写的吗？"
- **准确性优先**：事实、数据、逻辑必须和原文完全一致，重写不能重写掉准确性
- **自然语序**：用中文习惯的语序，长句拆成自然的短句；比喻和习语按意思翻译，不逐词直译
- **术语一致**：同一个术语全篇统一译法；专有名词首次出现时用"中文译名（英文原文）"的格式标注
- **保留格式**：标题、加粗、斜体、图片、链接、代码块等 Markdown 格式原样保留
- **主动补充解释**：目标读者可能缺乏背景的术语/概念，用**加粗括号**补充解释，例如"（**解释**）"；解释要精简，只在真正影响理解时才加

## 注解密度跟着读者画像走

**不要用固定的一套注解密度**——同一个术语，给初级工程师看要解释，给高级工程师看就是噪音。调用方（如 article-write）已有读者画像时，直接复用：

| 读者技术背景（来自读者画像） | 注解策略 |
|---|---|
| 无背景 / 初级工程师 | 术语首次出现必须加中文解释，常见缩写也展开一次 |
| 中级工程师 | 只解释该领域专属术语，通用工程术语（如 API、缓存）不加注 |
| 高级工程师 | 只在术语有歧义或该项目特殊用法时才加注，其余保持简洁 |

被独立调用（没有调用方提供读者画像）时，直接问用户目标读者的技术背景，不要假设。

## 常见误译信号（翻译时就要避免，不要留到审查阶段才发现）

翻译时如果写出以下几种结构，说明还在直译英文语序，需要重新组织成中文表达：

| 直译信号 | 例子 | 改法 |
|---------|------|------|
| 介词结构堆砌 | "通过...来..."、"对...进行..." | 去掉介词短语，用动词直接表达 |
| 因果结构倒置 | "X，是因为 Y"（照抄英文 because 从句语序） | 中文习惯先说原因，Y 放前面 |
| 名词化动词 | "进行压缩"、"实现了...的功能" | 直接用动词："压缩" |
| 被动语态过度 | "被...所决定"、"由...来完成" | 改主动语态 |

## 和 R2 的关系（强制，不可跳过）

这四类信号和 article-review 的 R2（翻译腔审查）用的是**同一套判断标准**，但两者不能互相替代：

- **翻译时自查**（上表）：省掉后续大部分返工，但翻译者很难发现自己写出来的翻译腔——这是认知盲区，不是态度问题
- **R2 独立审查**：翻译完成后**当场**（不等到全文写完）直接 Read `~/.agents/skills/article-review/references/r2-translation.md` 内联执行一遍。Claude 系模型生成中文时本身容易出现翻译腔和奇怪表达，尤其是刚读完英文源码/文档之后——这一步是质量的主要防线，不是可选的收尾动作

## 不需要的东西

不需要分块翻译、多种翻译模式（quick/normal/refined）、子 agent 并行翻译、EXTEND.md 配置文件这类工程化机制——本技能处理的是单篇文章素材量级的翻译，用不上这套基础设施。源文档特别长（远超单次会话上下文）时才考虑分段处理，优先级远低于翻译质量本身。
