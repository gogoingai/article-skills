# 自动发布扩展点（占位）

当前版本的 article-publish 只做本地文件生成（清洗正文 + 标题/简介/封面 + 写入 `publish/v{N}/`），**不做任何网络推送**。这份文档记录未来接入具体平台自动发布时的接口约定，目前没有实现，仅作占位说明。

## 设计意图

`meta.md`（见 `references/workflow.md` 第七步）就是为自动发布预留的接口——它把标题、简介、封面 URL、对应草稿版本这些平台通常需要的字段结构化落盘，未来接入某个平台时，自动发布脚本/技能只需要读这个文件，不需要重新解析文章正文。

## 接入新平台时需要考虑

- **认证方式**：大多数平台（公众号、掘金、个人博客 CMS）需要 API Key / Cookie / OAuth，应该作为独立的凭据配置，不写进任何 Markdown 文件（避免泄露）
- **格式转换**：`{文件名}.md` 是标准 Markdown，多数平台需要转换（公众号需要转 HTML 并处理图片单独上传，掘金支持 Markdown 但字段结构不同）——转换逻辑应该是平台专属的脚本，不污染 article-publish 的核心清洗逻辑
- **发布状态追踪**：接入自动发布后，`references/changelog.md` 的"发布"类型记录应该扩展记录发布到了哪些平台、平台侧的文章 ID/URL，方便后续更新同一篇文章时定位
- **风险提示**：真正推送到公开平台是不可逆操作（或撤回成本很高），实现自动发布时必须在推送前有一个明确的用户确认步骤（`AskUserQuestion`），不能因为本地产物已经生成就默认可以直接推送

## 当前占位行为

用户要求"自动发布到 XX 平台"这类当前未实现的能力时，如实告知"这个能力还没实现，目前只能生成本地发布版文件，你可以手动发布"，不假装执行、不编造发布结果。
