# 发布流程

> 本文件是 article-publish 技能的核心执行流程。

---

## 第一步：定位存储，确定当前版本

- 从用户提供的文章路径取文件名（不含扩展名），检查 `{项目根目录}/.article-skills/{文件名}/` 是否存在
  - **存在** → 读 `SKILL.md`（拿标识、写作目标等背景）和 `references/changelog.md`（拿当前最新草稿版本号 `vN`）
  - **不存在**（降级模式）→ 告知用户「没有找到本篇的 article-write 存储，将直接基于当前文件内容生成发布版，版本记为 v1」，后续步骤跳过所有依赖 `.article-skills/` 的读取，产物目录仍按下方规则创建（先建 `.article-skills/{文件名}/` 及必要子目录）

## 第二步：就绪检查

扫描全文，检查两类未完成标记：
1. `> 🖼️ 待配图：[描述]` 占位标记（还没转换成完整画图提示）
2. `# 画图提示` 代码块后面没有 `![]()`（提示写完了但图还没生成）

有任一类未完成项，用 `AskUserQuestion` 询问：

```
检测到全文还有 N 处配图未完成（M 处占位未转换 / K 处已有提示但图未生成）。

选项：
- 先去配图：先用 Skill 工具调用 article-image 补齐，再回来继续发布
- 直接发布，未完成的先留空：发布版对应位置留空白，不阻塞发布
- 取消：先不发布
```

用户选"先去配图" → 调用 article-image 补完后再回到本流程；选"直接发布" → 继续，清洗时这些占位/未完成提示原样按"留空"处理（见第三步）；选"取消" → 结束流程。

全部已完成（每处占位都已转换为提示且都已生成图片）时跳过本步的提问，直接进入第三步。

## 第三步：清洗正文

对文章全文做以下删除/替换，**不改变清洗范围之外的任何文字**：

1. **删除项目生成标识引用块**：H1 标题下方的 `> 🔖 本文由 [article-skills]... 标识 \`aid-xxxxxxxx\`...` 整段删除
2. **画图提示代码块只保留图片**：每个 `# 画图提示` 代码块（含其 YAML frontmatter）整段删除，只留下代码块之后的 `![标题](URL)`；第二步选择"直接发布，留空"的位置，代码块同样删除，不留任何提示痕迹（也不留空行占位的说明文字）
3. **占位标记未处理时**：`> 🖼️ 待配图：[描述]` 整行删除（用户已选择直接发布留空）
4. **扫一遍确认**：全文不再出现 `M\d+` 素材编号、骨架符号、`aid-` 标识、`# 画图提示`、`🖼️` 等内部标记；正文其余部分与草稿保持一致，不做额外改写

清洗结果先保存在内存/临时变量里，不直接覆盖原草稿文件——原草稿文件在整个发布流程中不应被修改。

## 第四步：生成标题

→ 查 `references/title-summary.md`

基于清洗后正文，生成 3~5 个候选标题，用 `AskUserQuestion` 展示（含清洗前的原 H1 标题作为兜底选项之一），选项形式：

```
{
  question: "为这篇文章选一个发布标题（或自己写一个）：",
  header: "发布标题",
  options: [
    { label: "候选1", description: "一句话说明这个标题的角度/亮点" },
    { label: "候选2", description: "..." },
    { label: "候选3", description: "..." },
    { label: "保留原标题", description: "沿用草稿的 H1：[原标题]" }
  ]
}
```

用户选"Other"手打自定义标题时直接采用。

## 第五步：生成简介

→ 查 `references/title-summary.md`

生成约100字（90~120字之间）的简介候选 1~2 个，用 `AskUserQuestion` 确认或接受修改意见，同上以选项形式展示候选全文（放入 `description` 或直接列出）。

## 第六步：生成封面图

用 `Skill` 工具调用 **article-image** 技能，传入 `references/cover-prompt.md` 定义的封面专属画图规范（不同于文中说明性配图，强调主题感/视觉冲击/适合做封面缩略图）。article-image 按其现有的生成/质检/`AskUserQuestion`确认流程完成，最终封面图 URL 不写回原草稿正文，只记入本次发布产物。

article-image 未安装或调用失败：告知用户「封面图生成失败/技能未安装，发布版将不含封面，可以后续手动补」，跳过本步继续。

## 第七步：写入产物

产出目录：`{项目根目录}/.article-skills/{文件名}/publish/v{N}/`（`N` = 当前草稿版本号；同一草稿版本重复发布时才会用到下面的发布次序号）

写两个文件：

**`{文件名}.md`**（发布版全文）：

```markdown
# [最终标题]

> [约100字简介]

![封面图](封面图 URL)

[清洗后的正文，从原来的 H2/H1 后续内容开始]
```

**`meta.md`**（供人工或后续自动发布脚本读取）：

```markdown
# 发布元信息

- 对应草稿版本：v{N}
- 生成时间：{时间}
- 最终标题：{标题}
- 候选标题（未选用）：{候选1}、{候选2}……
- 简介：{简介全文}
- 封面图 URL：{URL，若未生成则写"无"}
```

## 第八步：记版本

在 `{项目根目录}/.article-skills/{文件名}/references/changelog.md` 追加一行：

```
| v{N}-pub{M} | 发布 | {时间} | {最终标题} |
```

`M` = 该草稿版本号下第几次发布（同一 vN 反复发布 M 递增，不同 vN 各自从 pub1 开始）。

## 第九步：告知用户

汇报产物路径（`.md` 和 `meta.md` 的完整路径），以及本次是否有跳过的步骤（无封面/留空的配图位置等）。
