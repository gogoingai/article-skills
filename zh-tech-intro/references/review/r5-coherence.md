# R5：连贯性与清晰度（第一步跑）

**每次审查的第一步**：先整体读一遍，再进行逐条检查。连贯性和清晰度都要先发现——后面的 R1~R4 是在内容可读的基础上做的。

---

## 一、连贯性检查

**章节之间**
- 相邻章节是否有逻辑关联？结束一节时读者是否知道"接下来为什么要讲下一节"？
- 有没有章节突然切换话题，让读者感到"这怎么跳到这里了"？

**段落之间**
- 段与段之间有没有无铺垫的概念切换？
- 引入新概念前，前一段有没有为它铺垫？

**前后一致性**
- 某个机制/数字在文章多处提及，说法是否一致？
- 说了"有 N 种情况"的地方，是否实际讲了 N 种？

---

## 二、清晰度检查

连贯性是关于"流动"，清晰度是关于"读者能否在脑子里建立模型"。一节可以衔接顺畅，但读者依然看完一头雾水——这是不同的问题。

**逐节自问**：读者没有源码背景，读完这节能不能想象出它描述的东西长什么样？

**重点检查**：

| 场景 | 要求 |
|------|------|
| 多文件 / 多组件存储结构 | 表格之后必须为每个文件 / 组件给一段内容快照（代码块），让读者看到实际数据的形态 |
| 新机制首次出现 | 必须有一个具体场景或例子，不能只有抽象描述 |
| 流水线 / 管道式结构 | 必须说明数据的流向和每步的输入输出，不能只列步骤名称 |
| 抽象概念（置信度、评分、加权）| 给一个真实数字的例子，说明"达到这个分意味着什么" |

**判断标准**：删掉这节里所有的具体例子和数字，读者还能在脑子里画出这个机制吗？不能 → 缺清晰度。

---

## 常见信号汇总

| 信号 | 分类 | 处理方式 |
|------|------|---------|
| 读者读完这节，还不知道"接下来讲什么" | 连贯性 | 加过渡句 |
| 某个词/概念突然出现，前面没有铺垫 | 连贯性 | 往前加一句引入 |
| 前面说了"有三种情况"，后面只讲了两种 | 连贯性 | 补齐或删掉那句"三种" |
| 某节的开头和上一节的结尾完全没有关联 | 连贯性 | 加一句衔接 |
| 同一个机制在不同章节的描述有矛盾 | 连贯性 | 统一说法 |
| 多文件结构只有表格，没有内容快照 | 清晰度 | 为每个文件补代码块示例 |
| 新机制只有定义，没有具体场景或例子 | 清晰度 | 补一个端到端的具体例子 |
| 抽象评分/阈值没有"这意味着什么"的解释 | 清晰度 | 加一句"达到这个分说明……" |
| 读完整节无法想象它描述的东西长什么样 | 清晰度 | 补内容快照或具体场景 |

---

## 报告格式

- 发现问题 → 说明位置（章节名/段落）+ 类型（连贯性/清晰度）+ 问题描述 + 候选修法
- 未发现问题 → 一句话确认"连贯性与清晰度 clean"
