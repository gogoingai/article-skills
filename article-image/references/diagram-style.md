## 零、风格关键词触发

**写画图提示之前，先检查用户是否提到了风格关键词。** 匹配到关键词时，在 prompt 开头自动插入对应的风格模板片段，**同时**按下方规则决定是否传 `--ref`。

| 关键词 | 触发风格 | 模板位置 |
| --- | --- | --- |
| 手绘、插画、手绘风格、插画风格、手绘插画、像手画的、手绘感 | 手绘插画风格 | → §八，用「风格关键词」段的模板片段开头，**必须**加 `--ref` |
| excalidraw、白板手绘、白板草图、斜线填充、手绘线条图、像白板画的 | Excalidraw 风格 | → §十一，用「风格关键词」段的模板片段开头，**必须**加 `--ref`。注意与「手绘插画风格」的区别：Excalidraw 更工程感、无插画图标、斜线排线填充；手绘插画风格更可爱、有图标插画、实心色块填充 |
| 马克笔风格、单色手绘、波浪线注释、绿色手绘图、单色调手绘 | 单色马克笔风格（mono-marker） | → §十二，用「风格关键词」段的模板片段开头，**必须**加 `--ref`。与前两种手绘风格的区别：全图只用一种主色（节点、描边、箭头、分组框同色系），奶油色背景（非纯白），标志性的卷曲波浪线连接节点与旁注文字 |
| PPT、演讲风、幻灯片、技术PPT、slide、机器人风格 | 技术PPT风格 | → §九，用「prompt 风格描述」段的模板片段开头，**必须**加 `--ref` |
| （无关键词 / 默认） | 极简专业 PPT 风格 | → 按 §三 各图类型模板，结尾加「风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。」 |

### 手绘风格的生成规则（两者缺一不可）

**① `--ref` 优先**：传入参考图锚定视觉风格（有机线条、水彩填色感），仅靠 prompt 描述无法稳定复现。推荐参考图：

```
--ref ~/.agents/skills/article-image/assets/styles/handdrawn/handdrawn-01-skill-creation-loop.png
```

（也可换 `handdrawn-02` 至 `handdrawn-05`，选和目标图类型最接近的）

**② prompt 风格描述不可省略**：`--ref` 锁视觉质感，prompt 里的风格词控制模型对内容的理解方向，两者共同作用。prompt 开头必须加：

```
手绘插画风格，有机线条感，水彩填色质感，背景纯白。
```

**完整调用示例：**

```bash
RAND=$(openssl rand -hex 4)
bash ~/.agents/skills/gpt-image-2/scripts/gen.sh \
  --prompt "手绘插画风格，有机线条感，水彩填色质感，背景纯白。[图内容描述...]" \
  --ref ~/.agents/skills/article-image/assets/styles/handdrawn/handdrawn-01-skill-creation-loop.png \
  --out /tmp/zh-tech-img-${RAND}.png \
  --timeout-sec 300
```

参考图示例见 `../assets/styles/handdrawn/`，详细模板见 §八。

## 一、核心原则

### ⚡ 第一原则：描述视觉物体，不指定显示文字

模型在「画什么」和「写什么」之间有隐含的选择权——prompt 的写法决定它把预算花在哪里。

| 低效（指定文字） | 高效（触发元素） |
|---|---|
| 「三张卡片，分别写：快速响应、精准定位、低延迟」 | 「三张卡片，每张左侧带彩色图标，主题依次是闪电、靶心、时钟」 |
| 「右侧补充说明：该步骤负责过滤低质量输入，阈值 0.6」 | 「右侧灰色小注卡片，带盾牌图标，一句话质量门描述」 |
| 「底部备注区，文字说明 A、B、C 三种情况」 | 「底部图例区，三个色块 + 三个短标签，横向排开」 |

描述**视觉物体**（图标、色块、卡片、箭头）→ 模型画元素；描述**文字内容**（说明、注释、备注）→ 模型显示文字。

---

### 视觉效果描述优先于文字内容

| 多写 | 少写 |
|------|------|
| 整体氛围（「明亮通透」「色彩饱和但不杂乱」） | 节点副标签的详细文字 |
| 色彩对比（「图标是全图最鲜艳的元素，和浅色底色形成反差」） | 每张卡片的具体说明文字 |
| 视觉层次（「卡片是主角，箭头细而克制」「标题条比底色深一档」） | 过度精确的文字标注 |
| 空间感（「留白充裕」「两侧色块等高对齐」） | 渲染参数（颜色值、尺寸） |

---

### 绝对禁止

| 禁止 | 原因 |
| --- | --- |
| CSS 颜色值（`#3B82F6`） | 过度限制模型发挥，结果反而变差 |
| 像素/尺寸（`800px`、`14pt`） | 生图模型不接受绝对尺寸 |
| 字号/字重（`bold`、`font-size: 16`） | 渲染细节由模型决定 |
| SVG/HTML/CSS 代码片段 | 不是代码生成任务 |
| 箭头粗细、边框样式等渲染细节 | 描述内容和结构，不描述样式参数 |

### 应该写的内容

- **图类型**：横向多层架构图、竖向三阶段流程图、条形图+卡片……
- **节点标签**：全部用中文，描述语义（「同步决策」而非 `SyncDecision`）
- **箭头语义**：说明关系（「降级路由」「触发迁移」），不说方向
- **关键数据**：阈值、比例、数字直接写进来（「得分 0.62」「权重 30%」）
- **视觉层次**：「重点突出」「虚线框表示可选」「深色背景强调」

---

## 二、【禁用】泳道图

**不要使用纵向泳道图（Vertical Swimlane）。**

gpt-image-1 实践验证：纵向泳道图 prompt 生成结果极差——泳道线混乱、节点溢出边界、文字截断、整体像线框图而非正式图表。

**替代方案**：

| 原泳道图场景 | 替换为 |
| --- | --- |
| 跨模块的完整流程（跨角色） | 竖向三阶段流程图（§3.2） |
| 数据在多层/多组件间流转 | 横向多层架构图（§3.1）或横向阶段列流程图（§3.7） |
| 强调「谁做什么」的分工 | 横向阶段列流程图（§3.7），顶部加阶段标签 |

## 三、图类型参考模板

### 3.1 横向多层架构图

适用：展示系统分层结构、各层组件和整体数据流。  
参考图：`../assets/styles/other/01-architecture.png`（MemoryCore 四层架构，重点参考：左侧彩色层名徽章、节点图标+副标题、层间单箭头标语义）

**三区布局规范**（每行一层，左中右三区）：

| 区域 | 内容 | 视觉要求 |
| --- | --- | --- |
| 左区（层名徽章） | 层名文字 | 带颜色背景的圆角矩形，颜色比该行浅色背景深一档 |
| 中区（节点组） | 横向并排节点，每节点：小图标 + 加粗名称 + 灰色副标（1~2行） | 圆角矩形，浅色背景行 |
| 右区（角色描述） | 一句话说这层"是什么角色" | 灰色小字，垂直居中 |

**属性卡片 vs 流程节点**（最容易混淆，必须看清楚再写）：

| 情形 | 节点含义 | 写法 | 箭头 |
| --- | --- | --- | --- |
| 展示该层的**特征/属性** | 并排的属性描述（如：原始消息 / 顺序读取 / 会话级） | 「横向并排三张属性卡片，**卡片之间不用箭头连接**」 | ❌ 无 |
| 展示该层内部的**处理步骤** | 有先后顺序的操作（如：解析→提炼→落库） | 「三个步骤节点，箭头相连」 | ✅ 有 |

错误写法「内容三个节点串联」容易让生图工具在属性列表之间画出箭头，整层变成流程图而非存储架构图。

**层外数据流规范**：

架构图中层与层之间的数据流箭头（如「实时写入」「异步提炼」「混合检索」）必须明确声明在色带**外侧**，不能穿插在层内卡片之间。格式：

```
# 色带外部的数据流（均在色带轮廓外侧，不穿插在节点之间）：
#   顶部：「[来源]」→ 上方色带，标注「[操作]」
#   两层之间：居中垂直箭头，标注「[操作]」
#   右侧：[侧边线说明]
```

**层间箭头规范**：
- 每两层之间只画**一条**居中向下箭头，**不要扇形展开**连到每个节点
- 箭头标注具体操作语义（「调用」「读取 & 写入」「策略查询」），不写"数据流"这类泛称
- 箭头标签下方加白色矩形衬底，防止与背景混叠

```
# 画图提示：横向多层架构总览图，展示[系统名]的[N]层结构与数据流。
#
# 整体布局：[N] 行（每行一层）从上到下排列，每行分三区：
#   左区：层名徽章，带颜色背景的圆角矩形，颜色比该行背景深一档，层名居中
#   中区：节点横向并排，每节点由小图标 + 加粗名称 + 灰色副标组成
#   右区：一句角色定位描述，灰色小字
#
# 行 1【[层名]】（[浅色]背景行，左徽章[深色]）：
#   [N] 个节点，各带图标：
#     「[节点A]」  [副标题]
#     「[节点B]」  [副标题]
#   右区：[一句角色描述]
#
# 行 2【[层名]】（[浅色]背景行，左徽章[深色]）：
#   [N] 个节点，副标题为串联步骤：
#     「[节点C]」  [步骤1] → [步骤2] → [步骤3]
#   右区：[一句角色描述]
#
# 层间箭头（每层之间一条，垂直向下居中，不展开连到各节点，标签有白色衬底）：
#   [层名A] → [层名B]：「[操作语义]」
#   [层名B] → [层名C]：「[操作语义]」
#
# 风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。
```

### 3.2 竖向三阶段串行流程图

适用：有清晰阶段划分的处理流程，替代泳道图。

**标题栏位置：顶部横条（不是左侧竖条）**

每个阶段框的标题栏必须是**顶部横贯全宽的横条**，不要用左侧竖条。原因：左侧竖条挤占水平空间，步骤节点区域变窄，图整体显得拥挤。

```
# ✅ 正确：顶部横条
#   顶部深蓝横条标题栏（横贯矩形全宽），白色文字「[名称] · [副标题]」，左侧有图标；
#   标题条下方主体区域：…

# ❌ 错误：左侧竖条
#   左侧深蓝色竖条标题栏，白色文字「[名称] · [副标题]」；
```

**异步阶段标准表示**：

当某个阶段是**异步执行**（不阻塞主流程），需同时给出三个视觉提示：
1. 阶段框外轮廓改为**虚线**（橙色虚线）
2. 背景色选**浅黄**（与同步阶段的蓝/绿色系形成区分）
3. 标题栏**右端**加「**异步**」白色小胶囊标签

```
#   阶段二【[名称]】（浅黄背景，整行宽圆角矩形，整体外框为橙色虚线）：
#     顶部深橙横条标题栏（横贯矩形全宽）「[名称] · [副标题]」，标题栏右端有「异步」白色小胶囊标签；
#     标题条下方主体区域：…
#     主体最右侧有一行灰色小字「不阻塞主流程」
```

**步骤节点布局**：同一阶段内的多个步骤节点必须等宽、均匀分布，不要挤在一侧留大片空白。

```
# 画图提示：竖向三阶段流程图，展示[流程名]
#
# 顶部触发器：「[触发条件]」→ 判断菱形「[已完成检查]？」→ 是则结束，否则继续
#
# 三个串联阶段框（圆角矩形，从上到下）：
#   阶段一【[名称]】（浅蓝背景，整行宽）：
#     顶部深蓝横条标题栏（横贯全宽），左侧图标，白色文字「[名称] · [副标题]」
#     标题条下方：横向排列 N 个白色圆角步骤节点，等宽均匀分布，箭头相连：
#       「[步骤1]」→「[步骤2]」→「[步骤3]」
#   阶段二【[名称]】（浅黄背景，整行宽，整体外框橙色虚线）：
#     顶部深橙横条标题栏（横贯全宽）「[名称] · [副标题]」，右端「异步」白色小胶囊标签
#     标题条下方：横向排列 N 个步骤节点，等宽均匀分布，箭头相连
#     主体最右侧灰色小字：「不阻塞主流程」
#   阶段三【[名称]】（浅绿背景，整行宽）：
#     顶部深绿横条标题栏（横贯全宽），左侧图标，白色文字「[名称] · [副标题]」
#     标题条下方：左侧菱形「[判断节点]」，右侧引出 N 条等间距纵向分布的水平箭头，
#       各指向 N 张等高圆角卡片（纵向均匀排列）
# 阶段间：每两个阶段之间一条居中实线向下箭头
#
# 风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。
```

### 3.6 竖向串行流程图（含判断分支）

适用：单一路径的处理步骤，含条件判断分支。

**⚠️ 多旁路汇合时的四条强制规则**（经实践验证，缺失任一条都会出现连线错误）：

1. **旁路终点必须显式连入汇合节点**：每条旁路的最后一个节点，必须在 prompt 中明确写「→ 向下连入「[汇合节点名]」」，不能依赖模型自行推断。
2. **所有箭头居中进出**：在 prompt 中加一句「所有箭头从节点中央水平或垂直引出，进入目标节点中央」。
3. **多旁路纵向空间均匀分配**：有 N 条旁路时，在 prompt 中说明「N 条路径在图中纵向空间均匀分配，不要相互紧贴」。
4. **颜色一致性**：在 prompt 中加「所有节点配色保持同一色调体系，由模型自行决定，不要有的节点深色调有的浅色调」。

**⚠️ 禁止自动编号**：流程节点不加流水序号（①②③），除非明确要求。

```
# 画图提示：竖向流程图，展示[流程名]的处理步骤与分支逻辑。
#
# 从上到下：
#   「[入口节点]」（椭圆）
#     ↓
#   「[判断条件A]」（菱形）
#     是 →→ 右侧旁路：「[旁路A节点1]」→「[旁路A节点2]」→ 向下连入「[汇合节点]」
#     否 ↓

### 3.7 横向阶段列流程图

适用：多个处理阶段，每个阶段内有纵向子流程，阶段间横向推进。

**⚠️ 列间连接是最容易混乱的地方**，必须在 prompt 中逐一声明列间连线，不能只描述列内节点。  
核心规则：
- **每列的出口节点必须明确点名**连接到下一列的哪个入口节点。
- **列内分支终止时**（如某条件判断为"跳过"的终止路径），必须标注「此路径止于本列，不跨列」。
- **箭头颜色不要多于两种**，颜色多了模型会把线画乱；如需区分异步/同步，只用实线/虚线区分即可，不加颜色。

#     「[节点B]」
#     「[节点C]」
#   第③列（从上到下）：

#   第④列（从上到下）：

### 3.10 金字塔分层图

适用：展示层次递进关系，底层最基础/最宽，顶层最高级/最窄（如优先级金字塔、处理链路分层）。

**⚠️ 金字塔的四条强制规则**（经实践验证，违反任一条都会导致图形扭曲或层次混乱）：

1. **层序方向**：底层 = 第一层（最基础）= 最宽；顶层 = 最高层（最高级）= 最窄的塔尖。**编号从下到上递增**。模型默认容易画反，必须在 prompt 中明确标注「底部最宽为第一层，顶部尖顶为最高层」。
2. **角度必须一致**：三条规则—左侧斜边从底层左角到顶点是一条连续直线；右侧斜边从底层右角到顶点是一条连续直线；所有层的左右两侧均共用同一条斜边，不单独画。违反此规则会导致各层角度不一致，视觉极丑。
3. **必须有塔尖**：顶层要收成三角形的真实尖顶（apex），不是梯形。
4. **不加标题**：金字塔本身不加图表标题，除非用户明确要求。

```
# 画图提示：金字塔分层图，展示[N]层[主题]结构。
#
# 整体形状：等腰三角形，顶部收成尖顶（apex），无图表标题。
# 三角形两侧斜边从底层两角延伸至顶点，是贯通全图的连续直线，各层共用同一斜边。
#
# 从下到上共 [N] 层（底部最宽为第一层，顶部尖顶处为第[N]层）：
#
#   第一层（底部，最宽）：
#     层名标题（加粗）：「[名称]」
#     属性行（带小图标）：
#       📋 处理：[处理描述]
#       ⚙️ 操作：[操作描述]
#       ⏱️ 时间成本：低（[说明]）
#
#   第二层（中部）：
#     层名标题（加粗）：「[名称]」
#     属性行（带小图标）：
#       📋 处理：[处理描述]
#       ⚙️ 操作：[操作描述]
#       ⏱️ 时间成本：中（[说明]）
#
#   第三层（顶部，最窄，含塔尖）：
#     层名标题（加粗）：「[名称]」
#     属性行（带小图标）：
#       📋 处理：[处理描述]
#       ⚙️ 操作：[操作描述]
#       ⏱️ 时间成本：高（[说明]）
#
# 各层配色：三层分别用三种协调色（如蓝/橙/红渐变色系），颜色由模型自行决定。
#
# 风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。

- 两路必须用**不同背景色**区分（如浅蓝 vs 浅橙），顶部各有**深色标题栏**

### 3.12 左右结果对比图

适用：同一查询/输入在两种方案下的结果对比，重点突出命中 ✓ 与漏掉 ✗ 的差异（如关键词 vs 语义向量检索效果对比）。

与 §3.5 并排方案对比图的区别：§3.5 比较的是**流程路径**；§3.12 比较的是**结果列表**，每行结果带 ✗/✓ 图标和右侧小注。

**核心规范**：
- 顶部必须有**共用横幅**作为公共输入，横跨全宽
- 每行结果固定三段式：`图标（✗/✓）+ 内容 + 右侧灰色小注`
- 中央用**垂直灰色虚线**隔开两侧

### 3.13 两层流程图：主流程 + 展开细节

适用：一个步骤本身有几个子步骤，单独画一张图太单薄，但全塞进主流程一行又太挤——典型场景是"5步主流程，其中第4步还能再拆成4个子步骤"。比单独画两张稀疏的图density更高，比硬塞成一行8+卡片更不挤。

**核心规范**：
- 上层是主流程，横向卡片，其中一张"被展开"的卡片用更深的颜色/边框突出，小字写「展开见下方」
- 该卡片正下方引一条竖直箭头连到下层
- 下层是子步骤展开，用浅灰虚线边框的矩形框整体框住，左上角标小字说明这是哪一步的细节，框内卡片可以比上层卡片略小、颜色用同色系的浅一档变体
- 整张图依然要显式声明「横向宽屏构图，宽高比约16:9」（见踩坑14），否则上下两层会被压成竖版导致两层都拥挤

```
# 画图提示：横向宽屏构图，宽高比约16:9，禁止竖版，不要任何大标题文字，背景纯白。纵向两层流程图，风格为高级技术PPT配图，留白宽松不拥挤。
#
# 上层主流程：[N]个带图标的[颜色]圆角卡片从左到右水平排列，卡片间距宽松，用实线箭头连接。

```

## 七、常见踩坑教训

以下均为通用问题，图类型专属的规则已整合进 §三 各模板的 ⚠️ 段落。

### 踩坑 1：流程节点被自动加上流水编号

**现象**：节点前出现「1.」「2.」「3.」序号，视觉噪声大。  
**根因**：prompt 用有序列表格式描述节点，模型把序号当作标签的一部分渲染出来。  
**修复**：节点描述不加编号。用「→」表示顺序，不用「1. 2. 3.」。

### 踩坑 2：单个节点内容过多导致一侧留白严重

**现象**：某节点宽度撑大，左侧大面积留白，整体比例严重失衡。  
**根因**：把节点名 + 子分类枚举混在同一个节点里，宽度失控后左侧空旷。  
**修复**：子分类内容提炼为行内简短格式（「专有名词 / 复合名词 / 引用文本」一行平铺），不要和节点名混排。

### 踩坑 3：循环终止条件混入输出类型菱形

**现象**：把「超过最大轮次」作为第三条分支塞进「本轮输出属于哪一类？」的决策菱形，逻辑错误——LLM 只能输出「工具调用」或「最终结果」两种，「超过最大轮次」是循环计数器的状态，不是 LLM 的输出类型。  
**根因**：菱形节点的语义是「对同一事件的不同分支」，循环守门条件是另一件事。  
**修复**：循环终止条件单独提前为第一个菱形，在调用 LLM 之前先判断「是否超过最大轮次」；后面的分支菱形只负责判断输出类型，各自只有两条出路。

```
❌ 反例（菱形三出口）：       ✅ 正例（两个串联菱形，各司其职）：

进入循环                      进入本轮迭代
  ↓                             ↓
LLM 调用                      「是否超过最大轮次？」
  ↓                             ├── 是 → 返回空列表
「本轮输出哪一类？」              └── 否 → 调用 LLM
  ├── 工具调用 → 执行 → 循环           ↓
  ├── 最终结果 → 返回              「本轮输出哪一类？」
  └── 超过轮次 → 返回空 ❌          ├── 工具调用 → 执行 → 循环回入口
                                  └── 最终结果 → 返回
```

### 踩坑 4：只画极简骨架节点，图显得单薄单调

**现象**：节点只写一个动作短语，不带任何支撑数据，图沦为"过场插画"而非"信息载体"。  
**根因**：把画图提示当成章节标题大纲，只搬了步骤名，没有把关键数字、对比、设计取舍放进节点。  
**修复**：画图前检查对应正文段落，把以下三类内容塞进节点副标题/小字里：
- **关键数字**：阈值、权重、上限（「截断到 limit 条（默认 10）」）
- **对比/取舍**：两个版本的差异点（「初始分 0.58 vs 0.62」）
- **一句话设计动机**：为什么这步存在

**反向陷阱**：加密度不是把某些节点做成视觉主角、把其余降级成浮动注释——这样"信息有了，但关系没画出来"。优先保证每个节点视觉权重相同、箭头主线清晰，再考虑在副标题里补数字。

### 踩坑 5：横向多卡片图默认生成竖版画布，导致挤压拥挤（见 §3.13）

**现象**：描述"5个卡片横向排列"，但模型按竖版出图，卡片间距被挤没。  
**根因**：prompt 只描述排列方式，没有声明画布宽高比，模型用默认竖版。  
**修复**：横向多卡片/多列布局，prompt 开头必须加：「横向宽屏构图，宽高比约 16:9，禁止竖版」，同时写「卡片之间留白宽松不拥挤」。

### 踩坑 6：未声明背景色，模型给整图刷了大片强调色

**现象**：给某步骤起了"浅紫色卡片"，模型把强调色铺成整张图背景色。  
**根因**：描述了"用什么颜色"，但没声明"只能用在哪里"，模型选了影响范围最大的刷背景。  
**修复**：使用强调色时同时写两句——「整体背景必须是纯白色」＋「[颜色]只用在图标线条和强调色上」，两句缺一不可。

### 踩坑 7：色块颜色过深，整体压抑

**现象**：大面积深色色块，整张图沉重、阅读费力，和白色主背景割裂。  
**根因**：prompt 写了「深色背景」，模型忠实执行。  
**修复**：背景色块用浅色系，标题条比色块底色深一档即可。关键词：「明亮通透」「浅[颜色]背景，不要深色」。

### 踩坑 8：并列分支之间被误画出多余连接线

**现象**：一个判断节点分出两条并行、互不相关的分支（如「批处理」「实时处理」），模型会在两个分支节点之间额外画一条连接线/波浪线，把并列关系画成了顺序关系。分支水平间距越窄，越容易触发这个问题（尤其是当两个分支都要各自引一条波浪线做旁注时，模型容易把其中一条波浪线的起点/终点搭错到相邻分支节点上）。  
**根因**：prompt 只声明了"两条分支各自连什么"，没有显式声明"两条分支之间禁止连接"；分支间距过窄时，模型的空间布局也会本能地把相邻节点连起来。  
**修复**：两条缺一不可——① 在 prompt 里明确写「左右两条分支水平间距拉大（各占页面三分之一以上宽度），中间留白」；② 明确禁止句「[分支A节点] 和 [分支B节点] 之间禁止出现任何线条、箭头或波浪线连接」。多次实践验证：只写②不写①仍会复发，两句必须同时出现。

## 十、设计四大原则与配色审美

来源：《写给大家看的设计书》的亲密性/对齐/对比/重复四原则 + 色轮配色法 + 黄金分割构图。以下是转化为 **AI 生图 prompt 写法**的对应规则（不是让模型"理解设计理论"，而是把原则拆成可执行的 prompt 描述）。

### 4.1 亲密性——同类归组，异类拉开

同一语义组的节点必须挨在一起、间距一致；不同组之间用更大间距或背景色块分开。

| 已有体现 | 位置 |
| --- | --- |
| 三区布局（层名徽章/节点组/角色描述）左中右各自成组 | §3.1 |
| 两路并行图左右各自独立背景色包裹 | §3.11 |
| 手绘图各步骤用不同浅色底色区分 | §八 |

**写 prompt 时补一句**：「同组元素间距小而一致，组与组之间间距明显更大」。

### 4.2 对齐——看不见的网格线

同一层级的节点必须等宽/等高/顶对齐或中心对齐，不能错落随意摆放。

**已有强制规则**：§3.2「步骤节点必须等宽均匀分布」、§3.6「所有箭头居中进出」、§3.10 金字塔「所有层共用同一条斜边」。

**通用补充句**：多节点横排或纵排时，prompt 里加「同一行/列的节点顶部对齐，宽度一致」——不要只描述节点内容，不描述对齐关系。

### 4.3 对比——让重点跳出来

一张图只能有一个视觉焦点；次要信息必须在颜色饱和度、字号、边框粗细上明显降级，不能和主角平起平坐。

| 场景 | 对比手法 |
| --- | --- |
| 强调"待展开"的卡片（§3.13） | 颜色加深突出，其余卡片保持浅色 |
| 强调"异步阶段"（§3.2） | 虚线橙框 + 浅黄底，与同步阶段的实线蓝/绿框区分 |
| 强调图标（手绘风格，§八） | 图标是全图最鲜艳的元素，与浅色底色形成反差 |

**反向提醒**：对比不是"把每个节点都做鲜艳"——多个强调点互相抵消，等于没有重点（参考踩坑4的反向陷阱）。

### 4.4 重复——同类元素同一套样式

同一层级/同一类型的节点必须共享同一套视觉语言（配色体系、图标风格、圆角半径），不能一个节点方形一个节点圆形、一个深色一个浅色。

**已有强制规则**：§3.6「所有节点配色保持同一色调体系」、§3.10「各层配色由模型自行决定但三层需协调」。

**通用补充句**：「同类型节点复用同一种卡片样式（同圆角、同图标风格、同色系不同明度）」。

### 4.5 配色场景对照表

按画面里出现的颜色种类数选配色策略，不要凭感觉挑颜色：

| 场景 | 策略 | 写法示例 |
| --- | --- | --- |
| 2 种颜色对比（如 §3.5 方案对比、§3.12 结果对比） | 互补色，一主一强调 | 「左侧红色系（劣势方案），右侧绿色系（优势方案），互为对比色」 |
| 3 种颜色并列（如 §3.10 金字塔三层、§3.11 双路+汇合） | 等距三色组，愉悦不杂乱 | 「三层分别用蓝/橙/红渐变色系」（已在 §3.10 采用） |
| 多层/多组（4 种以上，如 §3.1 多层架构） | 同色系不同明度，不要彩虹跳色 | 「各层背景色统一为蓝色系，明度从上到下递减」而非「一层蓝一层绿一层紫一层黄」 |

**绝对避免**：任意堆砌 4 种以上互不相关的高饱和色——生图模型会画成"儿童拼图"而非专业配图。

### 4.6 黄金分割——已隐含在现有规则里，无需额外声明

- 横向宽屏图的 16:9 构图（§踩坑5）本身接近黄金比例（1.618:1），不需要再单独强调"黄金分割"。
- 左右分栏类图（§3.4、§3.11、§3.12）默认对半分栏即可；如果一侧是主逻辑一侧是辅助说明（图例/备注），可以按大约 6:4 分配空间而非严格对半，写法：「左侧主图区占约六成宽度，右侧说明区占约四成」。

### 4.7 完整感——以终为始（已有体现：踩坑4）

画图 prompt 之前先想清楚：**读者只看这张图、没有上下文，能不能看懂？** 检查清单：

- [ ] 图例/标注齐全——特殊符号（虚线、颜色、图标）如果超过一种含义，底部必须有图例说明
- [ ] 标题/子标题能让人一句话知道这张图讲什么（除非按 §3.13/§3.10 的规则明确不加标题）
- [ ] 关键数字/阈值/比例已经塞进节点副标题，不是空节点（踩坑4）
- [ ] 箭头方向和顺序无歧义（多旁路汇合已按 §3.6 四条规则显式声明连线）

## 十一、Excalidraw 风格参考库（excalidraw-style）

> 参考图存放于 `../assets/styles/excalidraw/`

与手绘插画风格（§八）容易混淆，但视觉语言完全不同——Excalidraw 风格更像工程师在白板上随手画的技术草图，没有插画图标，核心是"斜线排线填充 + 潦草描边 + 手写字体"。

| 特征 | 描述 |
| --- | --- |
| 线条 | 潦草粗糙描边（rough.js 风格），线条略微抖动不规整，非精确几何线，黑色为主 |
| 填充 | 斜线交叉排线（hachure fill），**不是**纯色实心填充——这是与手绘插画风格最大的区别 |
| 字体 | 手写体风格（类似 Excalidraw 默认 Virgil 字体），不用印刷体 |
| 图标 | **禁止**任何插画图标或卡通元素，只用最简单的几何形状（圆角矩形、菱形、圆形）+ 箭头 |
| 配色 | 柔和粉彩色，各区块/分组用不同颜色的排线区分，背景纯白 |
| 分组 | 大圆角矩形外框（细边框）套小节点，外框顶部或左上角标区块名 |
| 密度 | 工程感强调信息密度，多用嵌套分组表达系统分层，不留大片空白 |

## 十二、单色马克笔风格参考库（mono-marker-style）

> 参考图存放于 `../assets/styles/mono-marker/`

第三种手绘风格家族，与手绘插画风格（§八）、Excalidraw 风格（§十一）都不同——核心特征是**全图单一主色调 + 奶油色背景 + 波浪线引出注释**，视觉上更像用马克笔在米色笔记本上手绘的图解。

| 特征 | 描述 |
| --- | --- |
| 背景 | 米白色/奶油色（非纯白），带轻微纸张质感 |
| 配色 | **全图只用一种主色**（如绿色/蓝色）：节点描边、节点填充、连接箭头、分组外框全部同色系，只有极少数"类别标签"节点例外用灰色 |
| 节点 | 圆角胶囊/椭圆形，同色系描边+填充（填充可以是浅色调，不强制饱和实心） |
| 分组框 | 粗描边圆角矩形/胶囊形状，内部无填充（透明），把一组节点框在一起 |
| 线条 | 手绘感圆润流畅，不是锯齿状潦草描边（区别于 Excalidraw 的 rough.js 感） |
| 注释连接 | **标志性特征**：节点向右侧引出一条卷曲波浪线（形似手绘的 `~`），末端带箭头，连接到黑色手写说明文字；左侧/其他方向的说明用直线小箭头即可 |
| 字体 | 手写体，笔画偏粗饱满 |

**三种手绘风格家族的选择依据**：
- 用户提到「插画」「可爱」「卡通」「图标」→ §八 手绘插画风格
- 用户提到「excalidraw」「白板」「斜线填充」「工程草图感」→ §十一 Excalidraw 风格
- 用户提到「马克笔」「单色」「波浪线」→ 本节（§十二）

## 十三、该画哪类架构图

架构图按内容可分五类，每类回答不同的问题。写文章前先想清楚**这一节要回答哪个问题**，再决定画哪类图、用 §三 的哪个模板——不要一律套「多层架构图」了事。

| 架构图类型 | 回答的问题 | 关注点 | 对应 §三 模板 |
| --- | --- | --- | --- |
| 业务/产品架构图 | 系统解决什么业务问题？有哪些功能模块？用户怎么用？ | 业务闭环、功能分组（交互层/业务层/基础服务层/数据层） | §3.1（按功能层展示）或 §3.7（呈现业务流程） |
| 应用架构图 | 系统由哪些服务/组件构成？组件之间怎么调用？ | 服务拆分、调用关系、层间数据流 | §3.1、§3.7 |
| 技术架构图 | 某组件内部用了什么技术栈/中间件/部署方式？ | 非功能特征（高可用、扩展性），具体技术组件命名 | §3.1（作为最底层"基础设施"一行）、§3.2 |
| 代码架构图 | 模块/类/包之间的静态结构是什么？ | 代码分层（Controller/Service/DAO）、类关系 | 一般**不建议**用 AI 生图工具画——类图/包图这类精确结构图，用代码块或 mermaid 表达更准确，生图工具容易画错继承/依赖箭头的方向语义 |
| 数据架构图 | 数据怎么流转、存储、同步？读写策略是什么？ | 数据流向、分库分表、读写分离 | §3.7（数据流转阶段）、§3.11（如读写分离双路） |

### 金字塔层级关系

五类图不是并列关系，而是**系统级 → 应用级 → 模块级 → 代码级**的层层细化（上层包含下层）：业务/应用架构图属于系统级，技术架构图/数据架构图通常落在应用级或模块级，代码架构图属于代码级。技术介绍文章的典型结构因此是：

1. **开篇总览**：一张系统级图（业务架构图或应用架构图），让读者一眼看清"这是什么系统、由哪几块组成"——通常用 §3.1 横向多层架构图。
2. **各章节细节**：每讲一个机制，配一张模块级/应用级图（技术架构图或数据架构图），聚焦这一个机制内部怎么运作——用 §3.2/§3.6/§3.7 等流程类模板。
3. **代码级内容**：不单独画图，直接用代码块或 LaTeX 公式（见 R4 结构自检「概念性内容用画图提示或 LaTeX，不用代码块解释原理」这条针对的是公式/权重，纯代码结构仍用代码块）。

### 何时画

- 复杂系统的技术文章开篇必须有一张系统级总览图——没有总览图，读者会在细节里迷路。
- 不要等文章写完再补图：写到"这一段该配图了"的判断，应该在列大纲阶段就做，而不是写完文字后回头看哪里空了塞一张图。
