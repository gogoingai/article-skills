# 项目问卷生成指南

**必须使用 `AskUserQuestion` 工具交互式提问，不得把所有问题以文本一次性列出。**

---

## 整体流程

```
检查全局画像
    ↓
  存在 → 展示给用户确认 → 有修改则更新 → 进入文章专属问卷
  不存在 → 问前说明 → 建立全局画像 → 进入文章专属问卷
    ↓
文章专属问卷（扫系统后生成，扫描证据当场记入 materials.md）
    ↓
写入 SKILL.md（索引+关键信息）+ references/materials.md（素材库）+ references/preferences.md（偏好与反馈库，随时追加）
```

---

## 全局画像

### 路径（优先级从高到低）

| 优先级 | 路径 | 说明 |
|--------|------|------|
| 1 | `.gogoingai/article-write/profile.md`（项目目录） | 项目级覆盖 |
| 2 | `$HOME/.gogoingai/article-write/profile.md` | 用户全局 |

通常写在用户全局路径，只有特殊项目才用项目级覆盖。

### 检查与加载

**有全局画像时**：把画像内容展示给用户，一句话说明它的作用，然后用 `AskUserQuestion` 确认：

> 「我找到了你之前保存的读者/作者画像，用它来规划这篇文章会更准确。请确认它是否还适用：」
>
> 展示画像内容（读者背景、作者背景、语言偏好）
>
> 然后问：「这个画像还准确吗？」
> - A) 完全准确，直接用
> - B) 大部分准确，需要小改
> - C) 需要重新建立

- **A** → 直接进入文章专属问卷
- **B** → 用 `AskUserQuestion` 只问需要修改的部分，更新画像文件
- **C** → 走完整建立流程

**没有全局画像时**：先说明再建立（见下方"问前说明"）。

### 格式

```markdown
# 读者/作者画像

## 典型读者背景
| 维度 | 程度 |
|------|------|
| 技术背景 | [无 / 初级工程师 / 中级工程师 / 高级工程师] |
| AI/LLM 熟悉度 | [不了解 / 知道概念 / 用过工具 / 深度使用或开发] |
| 分布式/系统设计 | [不了解 / 知道基础 / 有经验] |
| 阅读目的 | [了解系统 / 学习设计 / 技术选型 / 深度研究] |
| 读完后的期望结果 | [知道系统能做什么 / 能判断是否值得研究 / 能上手使用 / 能参与开发] |

## 作者技术背景
| 领域 | 熟悉程度 |
|------|---------|
| [领域1] | [了解概念 / 实际使用 / 深度开发] |
| [领域2] | ... |

## 写作动机
- 典型场景：[对外推广 / 内部沉淀 / 技术分享 / 个人总结]
- 发布渠道：[个人博客 / 公司博客 / 团队内部文档 / newsletter / 演讲稿]

## 默认偏好
- 语言：[中文 / 英文]
- 文章深度：[功能层 / 设计层 / 实现层]
- 文章模式：[A 快速入门 / B 架构深挖 / C 方案对比]
```

## 问前说明

**没有全局画像时，建立画像前必须先说明：**

> 「在开始规划这篇文章之前，我想先了解一下你的典型读者是谁、你自己的技术背景是怎样的。
>
> 这些信息会保存成一个画像文件，以后写其他文章时直接加载，不用重复回答。
>
> 我会分几轮问几个问题，每轮 2~3 题，回答完就开始规划。」

然后用 `AskUserQuestion` 开始提问。

## 文章专属问卷（扫系统后生成）

全局画像确认/建立完后，**先用 Grep/Read 扫该系统，找到 3~5 个核心机制各自的具体文件和行号**，再生成 **6~10 个系统专属问题**，分 2~3 轮用 `AskUserQuestion` 问完。

**扫描是硬性前提，不是可选步骤**：不能凭项目名或 README 一句话描述就出题。扫完确实找不到某个机制的实现，如实说明「扫了 [目录]，没找到 [机制] 的实现，当作待确认处理」，不要假装扫过。

**扫到的每条证据当场追加进 `references/materials.md` 的『源码证据』表**（格式见文末「写入 materials.md」一节），不要等问卷问完、骨架都要出了才回头凭记忆补——扫描和记录是同一个动作。

**问题维度：**

### A0. 背景深挖（用户主动提供背景信息时必做，通用判定能力，不是固定模板）

**这是一种判定能力，不是绑定某个场景的固定问题清单。** 不要因为素材是"改写已发布文章避免查重"就套用一组写死的选项——换成"内部资料要对外发布""老文章要翻新""竞品最近发了类似内容所以要调整角度"，问题的具体内容都会不一样。判定标准只有一条：

> **用户主动说出的每一句背景信息，背后通常还有没说出口的"为什么"和"卡在哪"。听到背景信息只是记下来，不等于理解了背景。**

**识别信号**（出现任意一种，就要深挖，不能只当参考资料收下）：
- 用户提到"这个之前发过/写过/做过"——判断标准是"发过"这件事本身会不会限制这次的写法（读者会不会有印象、渠道是否相同、时间隔了多久）
- 用户提到明确的外部要求（"公司要求""领导让我""这是任务"）——判断标准是这个要求背后有没有硬约束（deadline、必须包含的内容、不能触碰的红线），用户往往只说了要求本身，没主动说约束
- 用户提到"避免 XX"（避免重复、避免太深、避免太长）——判断标准是"避免"针对的是谁的感受（读者会不会看腻、审核会不会打回、平台会不会判重），不同对象决定不同应对策略
- 用户提供的素材本身不完整（"梦境部分我后面发你""这块先空着"）——判断标准是缺失的这部分是可选内容还是这篇文章成立的必要条件；如果是必要条件，deadline 逼近但素材未到位时该怎么办，必须现在就问清楚，不能拖到真遇到才慌

**深挖的做法**：识别到上述信号后，**现场根据这句背景信息本身生成 1~3 个针对性问题**（用 `AskUserQuestion`），问题和选项都要贴着这句话的具体内容写，不能套用别的场景的选项。以下三个方向可以作为组织问题时的检查清单（不是必须逐条问，而是判断"这句背景信息里，这三个维度分别答清楚了没有"）：

1. **动机/为什么**：这件背景事实存在的原因是什么？会不会影响文章该写成什么样？
2. **硬约束**：有没有回避不了的边界条件（时间、渠道、必须/不能包含的内容）？
3. **未决风险**：如果某个前提没兑现（素材没到、时间不够、约束有冲突），谁来做决定——是现在就问清楚"到时候怎么办"，还是可以先搁置到真正发生时再问？**能现在问清楚的，不要拖到执行阶段才发现要靠自己拍板。**

回答写入 `SKILL.md` 最前面新增的 `## 背景` 字段（见下方模板），字段内容根据这次深挖出来的具体维度组织，不套固定表格列。深挖过程中用户主动甩出的具体素材（截图要点、原文片段等）额外按条目追加进 `references/materials.md` 的『用户提供的素材』表。

### A. 读者对该系统核心概念的熟悉程度（3~5 题）

针对这个系统最重要的 3~5 个概念各出一题，**问之前先引用扫描找到的 `path:line`**，选项统一为：
- 没听说过 / 知道大概 / 用过接触过 / 深入了解

例：不问"你对缓存机制熟悉吗"，改问"`cache.py:42` 用的是 LRU 淘汰策略，你对这个策略熟悉到什么程度？"——引用具体证据比泛泛提概念名更容易让用户校准回答。

### B. 本篇写作目标（3~4 题）

- **写作动机**：这篇文章为什么要写？和你的全局典型场景有什么不同？
- **读者起点**：读者读这篇文章之前，对这个系统了解多少？（完全不知道 / 听说过名字 / 知道能做什么 / 用过但不了解原理）
- **读完期望**：读者读完这篇文章，你希望他们能做到什么？（可能和全局偏好不同）
- **文章模式**：侧重哪个方向？（A 快速入门 / B 架构深挖 / C 方案对比）
- **发布渠道**（可选）：会发布在哪里？（影响术语解释密度和语气）

### C. 作者对该系统各模块的了解（3~4 题）

针对该系统主要模块各出一题，选项统一为：
- 需要你来研究 / 大致了解 / 比较清楚 / 读过源码

### D0. 范围锁定（必问，骨架前对齐，不可省略）

这一节的目标：在提骨架之前锁定"这次不写什么"，防止骨架阶段贪多、写到一半才发现摊子太大要返工。

```
AskUserQuestion({
  questions: [
    {
      question: "这篇文章明确不覆盖哪些模块/细节？（基于扫描到的模块列表）",
      header: "范围排除",
      multiSelect: true,
      options: [
        // 逐个列出扫描到但明显和本篇写作目标无关的模块/细节
        { label: "[模块/细节 A]", description: "[为什么可以排除]" },
        { label: "[模块/细节 B]", description: "[为什么可以排除]" },
        { label: "全部都要覆盖，不排除", description: "" }
      ]
    },
    {
      question: "如果篇幅要砍半，拿掉哪些内容这篇文章仍然成立？",
      header: "最小可行版本",
      options: [
        // 基于系统核心机制列选项，标出"拿掉后文章还立得住"的那部分
        { label: "只保留 [核心机制 1/2]", description: "其余标记为可选，视篇幅决定" },
        { label: "都是核心，砍不动", description: "" }
      ]
    }
  ]
})
```

回答写入 `SKILL.md` 的 `## 范围锁定` 字段。骨架阶段如果想加回被排除的模块，先回来跟用户确认，不要默默加回去。

### D. 内容策略（必问，骨架前对齐，避免反复改稿）

这一节的目标：在提骨架之前把"怎么写"对清楚，防止骨架提完再一轮一轮改深度、改例子、改结构。

**D1. 各主要章节的期望深度**

针对该系统的主要章节（扫系统后确定），逐章问期望深度：

```
AskUserQuestion({
  questions: [
    {
      question: "【章节名】这一章你希望写到哪个层次？",
      header: "章节深度",
      options: [
        { label: "简写带过", description: "说清楚是什么、做什么，不展开细节" },
        { label: "设计层", description: "重点说为什么这样设计，含主要取舍，跳过公式" },
        { label: "实现层", description: "展开参数、公式、源码逻辑" },
        { label: "视内容而定", description: "整体设计层，但某些核心机制展开到实现层" }
      ]
    }
    // 每个主要章节一题，合并在一次 AskUserQuestion 调用里（最多 4 题一组）
  ]
})
```

**D2. 例子策略**

```
AskUserQuestion({
  questions: [
    {
      question: "全文用一个贯穿始终的主线例子，还是每章用最合适的独立例子？",
      header: "例子策略",
      options: [
        { label: "单一主线例子", description: "同一个场景从头跟到尾，读者不用切换上下文" },
        { label: "章节独立例子", description: "每章选最能说明该机制的例子，互不关联" },
        { label: "主线 + 局部补充", description: "大框架用主线，某些机制用更合适的局部例子" }
      ]
    },
    {
      question: "你有偏好的例子场景吗？（比如：代码重构、测试规范、部署流程、团队协作规则……）",
      header: "例子偏好",
      options: [
        { label: "你来建议", description: "根据系统特性选最合适的" },
        { label: "有偏好，见文字输入", description: "" }
      ]
    }
  ]
})
```

**D3. 复杂内容的处理偏好**

```
AskUserQuestion({
  questions: [
    {
      question: "遇到复杂内容（公式、多步流程、抽象机制），优先用哪种方式帮读者理解？",
      header: "复杂内容处理",
      multiSelect: true,
      options: [
        { label: "多画图", description: "流程图、漏斗图、时序图辅助" },
        { label: "多举例", description: "每个抽象机制配一个具体数字例子" },
        { label: "直觉解释优先", description: "先说白话，再给公式" },
        { label: "完整推导", description: "公式来源和计算过程全部展开" }
      ]
    },
    {
      question: "多个章节涉及相似机制时（比如轻度和 REM 都有梦境日记），怎么处理重复内容？",
      header: "重复内容处理",
      options: [
        { label: "第一次详写，后面一句话带过", description: "「同上，不重复」" },
        { label: "每次都写，但侧重差异", description: "强调和前面有什么不同" },
        { label: "合并成一节统一讲", description: "两个地方共用一节" }
      ]
    }
  ]
})
```

以上问题回答完后，将结果写入 `SKILL.md` 的 `## 内容策略` 字段，再开始出骨架。

### E. 补充（必问，最后一轮单独问）

```
AskUserQuestion({
  questions: [
    {
      question: "还有什么要补充的吗？比如：某个模块特别想深入、想避开的内容、之前写过类似文章的经验、对这篇文章有什么特别的要求。",
      header: "补充说明",
      options: [
        { label: "没有，按上面的答案写就好", description: "" },
        { label: "有，见下方文字输入", description: "选这个后在文字框里补充" }
      ]
    }
  ]
})
```

用户选"没有"则跳过；选"有"或直接输入文字，则将内容写入 `SKILL.md` 的 `## 特别说明` 字段。

## 写入 SKILL.md

系统专属问卷回答完后，把结果写入 `{项目根目录}/.article-skills/{文件名}/SKILL.md`（项目根目录 = `git rev-parse --show-toplevel`，不是 git 仓库则用当前工作目录；文件名 = 本篇文章的文件名，不含扩展名）。

**SKILL.md 必须自包含**：全局画像当时确认好的内容要**整段拷贝**进下面模板的「读者画像」一节，不是写一句"参见全局画像"就跳过——本篇的写作依据不应该依赖全局画像文件之后还存在、还没被改动。

```markdown
# 写作上下文

## 索引
- [素材库](references/materials.md)
- [骨架](references/skeleton.md)

## 背景
[仅当 A0 深挖出实质内容时才写这一节，没有背景信息可挖就省略。内容不套固定表格列，按 A0 挖出的具体维度组织，通常包括：
- 背景事实本身（用户说了什么）
```

## 写入 preferences.md

偏好与反馈库落在 `{项目根目录}/.article-skills/{文件名}/references/preferences.md`，存放"这次说的话，下次同类操作还用得上"的持久化反馈——写法习惯、画图风格、术语取舍、审查侧重、翻译腔判定尺度等。**不再向用户提议修改技能仓库文档**，可复用的反馈一律记进这里，跟着这篇文章走。
